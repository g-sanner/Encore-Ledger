## Functional Requirements

### Transaction Management
- User can manually enter a transaction
- User can edit or delete existing transactions
- Transactions are stored persistently and locally
- User can import transactions via CSV files retrieved from bank
- Transactions should not be able to be imported twice

### Category Management
- User is provided with a list of common categories (ie, gas, grocery, utilities)
- User can create custom categories
- User can edit or delete existing categories
- Every transaction should be assigned to one and only one category
- Categories should not be able to be deleted if there are still transactions associated with that category

### Reporting
#### Personal Reporting
- Programs generate both daily and monthly summaries
- Summaries include
  - Total spending
  - Spending by category
- User can export data in a structured format
- Program prepares user data so the legacy functions can analyze it and generate reports

#### Business Reporting
- Program can generate Income Statements over a specific time period.
- Program can generate Balance Sheets as of a specific date.
- Program classifies business transactions into accounting-standard roles:
  - Revenue
  - Costs of Goods Sold
  - Assets
  - Liabilities
- Program validates that required roles exist before generating reports.

### Legacy System Integration
- System prepares data to pass to the legacy program
- System interfaces with the executable COBOL file locally.
- System displays the report generated by the legacy program to the user.
- Any errors from the legacy system are captured and displayed as readable messages.

## Non-Functional Requirements

### Installation
- Application runs locally on Windows machines
- Setup instructions are available in README

### Security
- All financial data remains local to the user's machine
- NO external API calls
- Sensitive operations, such as data deletion, requires confirmation
